Этот файл ведётся в соответствии с требованиями ТЗ: задокументировать случаи,
когда принятое решение расходилось с предложением Claude.

## #1 — Менеджер зависимостей: pip → Poetry

**Что предложил Claude:**  
Использовать `pip` с `pyproject.toml` (стандартный подход без явного менеджера).

**Почему нет:**  
pip не создаёт lock-файл с точными версиями транзитивных зависимостей. При установке
на другой машине или в Docker возможны расхождения версий, которые трудно воспроизвести.

**Что сделал вместо:**  
`Poetry` — генерирует `poetry.lock`, команда `poetry install` воспроизводит
окружение побайтово. Lock-файл коммитится в git.

---

## #2 — Масштаб планирования: всё сразу → модуль за модулем

**Что предложил Claude:**  
Спроектировать и начать реализацию всей системы сразу: Scoring Module + Market Module +
Eval Harness + деплой — в рамках одного планирования и одного потока задач.

**Почему нет:**  
Система состоит из двух независимых модулей с разными источниками данных, разными
интерфейсами и разными метриками качества. Попытка держать весь объём в голове одновременно
увеличивает когнитивную нагрузку и переполнение контекстного окна. Каждый модуль заслуживает
отдельного цикла: диаграмма - планирование - ревью плана - декомпозиция - реализация sub-agents - ревью.

**Что сделал вместо:**  
Последовательная декомпозиция: сначала полностью проектируем и реализуем **Scoring Module**,
затем переходим к **Market Module**. Использую superpowers в качестве принуждения следовать правилам.

---

## #3 — Порядок работы: файловая структура вместо первой диаграммы

**Что предложил Claude:**  
Сразу дерево файлов и директорий (`app/core/scoring/engine.py`, и т.д.) как основу систему. Акцент смещён 
в сторону кода а не логики системы. 

**Почему нет:**  
Файловая структура — это следствие архитектуры, а не её источник. Если начать с файлов,
граница ответственности модулей и потоки данных между ними остаются неявными.
Когда начинаешь кодировать, легко нарушить изоляцию просто потому что так проще расположить.

**Что сделал вместо:**  
Сначала draw.io-диаграмма взаимодействий (`docs/scoring-module.drawio`) — участники,
потоки данных, входы и выходы каждого модуля. Структура файлов будет выведена из неё.
